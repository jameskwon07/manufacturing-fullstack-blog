> 이 글은 특정 회사의 업무나 내부 정보를 다루지 않고, 제조 도메인 풀스택 개발을 공부하려는 주니어 개발자를 위해 작성한다.

## 1. 풀스택 개발은 도메인에 따라 완전히 달라진다

- 같은 로그인, 목록, 상세, 수정 화면이라도 어떤 산업에서 쓰이는지에 따라 중요한 결정이 달라진다
- 일반 웹 서비스에서는 사용자 경험과 빠른 실험이 중요할 수 있다
- 제조 시스템에서는 정확성, 추적성, 권한, 이력, 안정성이 더 중요할 수 있다
- 그래서 풀스택 개발자는 기술 스택뿐 아니라 업무 맥락을 함께 이해해야 한다

## 2. 제조 현장의 소프트웨어는 현실 세계와 연결된다

- 화면 속 데이터가 실제 작업, 설비, 자재, 검사, 품질 판단과 이어진다
- 잘못된 입력 하나가 단순한 화면 오류를 넘어 업무 혼선으로 이어질 수 있다
- 사용자는 개발자가 아니라 작업자, 관리자, 품질 담당자, 생산 담당자일 수 있다
- 소프트웨어는 현장의 실수를 줄이고 판단을 빠르게 만드는 도구가 되어야 한다

## 3. 제조 앱에서 자주 만나는 문제들

- 작업지시를 어떻게 만들고 진행 상태를 관리할 것인가
- 설비, 작업자, 자재, 공정 데이터를 어떻게 연결할 것인가
- 검사 결과와 불량 이력을 어떻게 남길 것인가
- LOT 또는 Serial 단위로 이력을 어떻게 추적할 것인가
- 엑셀로 관리하던 업무를 시스템으로 옮길 때 무엇을 먼저 바꿀 것인가
- 누가 어떤 데이터를 수정할 수 있고, 변경 이력을 어떻게 남길 것인가

## 4. 제조 도메인에서 풀스택 역량이 중요한 이유

- 프론트엔드: 현장 사용자가 빠르고 정확하게 입력할 수 있는 화면
- 백엔드: 업무 규칙, 상태 변화, 권한, 검증 로직
- 데이터베이스: 작업지시, 공정, 설비, LOT, 검사, 불량, 이력의 관계
- 운영: 로그, 감사 이력, 장애 대응, 배포와 복구

## 5. 구현보다 먼저 고민해야 할 설계 질문

- 이 기능은 어떤 업무 문제를 줄이기 위한 것인가
- 이 데이터는 누가 만들고, 누가 확인하고, 누가 수정할 수 있는가
- 상태는 어떤 순서로 바뀌며, 되돌릴 수 있는가
- 예외 상황이 생기면 현장에서는 어떻게 처리하는가
- 나중에 감사, 품질, 추적을 위해 어떤 이력이 필요한가
- 화면보다 먼저 데이터 흐름과 업무 흐름을 설명할 수 있는가

## 6. 주니어 개발자가 먼저 공부하면 좋은 것

- 제조 기본 용어: 작업지시, 공정, BOM, LOT, 검사, 불량, 설비, 재고
- 데이터 모델링: ERD, 상태 전이, 이력 테이블, 감사 로그
- 업무형 UI: 테이블, 필터, 검색, 대량 입력, 엑셀 업로드/다운로드
- 백엔드 설계: 권한, 검증, 트랜잭션, 배치, 알림
- 운영 기본기: 로그, 모니터링, 백업, 장애 대응 문서

## 7. 포트폴리오로 만들기 좋은 프로젝트

- 미니 MES: 작업지시, 공정 진행, 검사 결과, 불량 등록, LOT 추적, 대시보드
- 설비 점검 관리 앱: 점검 항목, 담당자, 주기, 이상 보고, 조치 이력
- 품질 검사 기록 앱: 검사 기준, 결과 입력, 불량 유형, 재검사, 통계
- 재고 추적 앱: 입고, 출고, 위치 이동, LOT 추적, 재고 조정
- 생산 현황 대시보드: 작업 진행률, 불량률, 설비 상태, 지연 작업

## 8. 정리

- 제조 도메인 풀스택은 단순히 화면과 API를 만드는 일이 아니다
- 사람, 장비, 자재, 공정, 품질 데이터를 하나의 시스템으로 연결하는 일이다
- 주니어 개발자에게는 낯설지만 데이터 모델링과 설계 감각을 키우기 좋은 분야다
- 처음부터 모든 것을 알 필요는 없고, 작은 제조 업무 하나를 정해서 흐름과 데이터를 그려보는 것부터 시작하면 된다
